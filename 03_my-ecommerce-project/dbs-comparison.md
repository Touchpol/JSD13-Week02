# Database Comparison: Why MongoDB for SnapPlay?

ในโปรเจกต์นี้ เราเลือกใช้ **MongoDB** (NoSQL) แทนที่จะเป็น Relational Database (SQL) สำหรับการจัดการบริการที่หลากหลาย ด้วยเหตุผลหลักดังนี้:

## 1. Schema Flexibility (ความยืดหยุ่นของข้อมูล)
บริการแต่ละประเภทบน SnapPlay มีรายละเอียดแตกต่างกัน:
- **จองช่างภาพ:** ต้องการข้อมูล Portfolio, วันเวลาว่าง, สถานที่, ประเภทการถ่ายภาพ
- **เช่าเลนส์:** ต้องการข้อมูลสเปกเลนส์, ราคาประกัน, จำนวนวันเช่า, ชนิดเลนส์ที่ส่งผลต่อราคา
- **ซื้อเวลาสตรีมเมอร์:** ต้องการข้อมูลแพ็กเกจชั่วโมง, เกมที่เล่น, แพลตฟอร์ม, ตารางเวลาว่าง
- การใช้ MongoDB ช่วยให้เราเก็บข้อมูลเหล่านี้ใน Collection เดียวกันโดยไม่ต้องมี Null fields หรือ JOINs ที่ซับซ้อน

## 2. Embedded Documents for Order Flow (จัดการข้อมูลคำสั่งซื้อ)
คำสั่งซื้อแต่ละประเภทมีการคำนวณราคาแตกต่างกัน:
- **Lens Rental:** ราคาคำนวณจากวันที่เช่า, ชนิดเลนส์, วันที่คืน, และระยะเวลา (price_per_day × rental_days) พร้อมส่วนลดรายสัปดาห์/รายเดือน
- **Photographer Booking:** ราคาคำนวณจากเรทช่างภาพต่อชั่วโมง × จำนวนชั่วโมง
- **Streamer Session:** ราคาคำนวณจากแพ็กเกจชั่วโมงที่เลือก
- **Nested Arrays:** เก็บ timeline การเปลี่ยนแปลงสถานะไว้ใน document เดียวกัน
- **Dynamic Pricing:** สามารถคำนวณราคาตามจำนวนวัน/ชั่วโมง และเก็บเป็นโครงสร้างที่ยืดหยุ่น

## 3. Rapid Feature Evolution (พัฒนาฟีเจอร์ใหม่ได้เร็ว)
เนื่องจาก SnapPlay เป็น Marketplace ที่มีบริการหลากหลาย เราต้องการ:
- เพิ่มประเภทบริการใหม่ได้โดยไม่ต้อง Migrate Database
- ปรับเปลี่ยนฟิลด์ตามความต้องการของตลาดได้ทันที
- รองรับ JSON structure ที่แตกต่างกันในแต่ละบริการ (เช่น ราคาเช่าเลนส์ VS ราคาจ้างช่างภาพ VS ราคาเล่นเกม)

## บทสรุป
MongoDB ตอบโจทย์ที่สุดสำหรับระบบ Marketplace ที่ต้องการความรวดเร็วในการพัฒนา (Agility) และรองรับความหลากหลายของประเภทบริการ ซึ่งเป็นหัวใจสำคัญของ SnapPlay
